Active Directory Environment

This lab demonstrates the deployment and administration of a simulated enterprise Active Directory environment using Windows Server 2022. I created a fictional organization called Company.local to mirror how small and medium-sized businesses manage centralized identity, authentication, security policies, and client computers. Throughout this project I deployed Active Directory Domain Services (AD DS), designed an Organizational Unit structure, created users and security groups, configured Group Policy Objects (GPOs), managed DNS and DHCP, and joined Windows clients to the domain.

Server Deployment & Domain Controller

I began by deploying a Windows Server 2022 virtual machine and installing Active Directory Domain Services (AD DS). After installation, the server was promoted to a Domain Controller by creating a new Active Directory forest named company.local.

Creating a dedicated forest establishes the foundation for centralized authentication, authorization, and identity management. In enterprise environments, every user, computer, and security policy depends on the Domain Controller.

Organizational Unit Design

Instead of placing every user inside a single container, I created separate Organizational Units (OUs) for each department.

This structure reflects how enterprise environments organize users. Department-based OUs simplify administration, allow Group Policies to be targeted more efficiently, and make delegated administration easier as organizations grow.

Each user was also configured with "User must change password at next logon", allowing employees to establish their own secure password while preventing administrators from knowing permanent credentials.

User & Security Group Management

After creating departmental users, I organized permissions using Security Groups. Rather than assigning permissions directly to individual users, enterprise administrators typically assign permissions to groups, then place users inside those groups.

This approach follows Role-Based Access Control (RBAC), making administration more scalable and reducing future maintenance.

Users were then assigned to the appropriate departmental groups, allowing access permissions to be inherited automatically.

Group Policy Configuration

Group Policy allows administrators to centrally configure security, desktop settings, and operating system behavior without manually configuring every computer. In enterprise environments, this ensures every device follows consistent security standards.

User Configuration

Applies policies directly to user accounts regardless of which computer they log into.

Computer Configuration

Applies policies to the device itself, regardless of which user signs in.

Policies vs Preferences

Policies enforce security settings, while Preferences provide default configurations that users may later modify.

Lab Implementation

To demonstrate centralized user management, I configured a User Configuration policy that prevents users from opening Control Panel. This represents a common enterprise restriction used to prevent unauthorized system changes.

Prohibit access to Control Panel

I also configured an Account Lockout Policy under Computer Configuration to automatically lock user accounts after multiple failed login attempts. This helps reduce password guessing attacks and improves overall account security.

Account Lockout Policy

Policy Verification & Testing

After configuring the Group Policies, I forced the client computer to retrieve the latest policies from the Domain Controller using:

gpupdate /force gpresult /r

The policy refresh completed successfully and gpresult confirmed that both the User Configuration and Computer Configuration policies were being applied to the client computer.

GPO in Practice

Simply creating a Group Policy is not enough—it must be verified on the client computer.

After updating Group Policy, I signed into a domain-joined workstation using a standard user account. When attempting to open Control Panel, access was successfully blocked by the policy configured earlier.

This demonstrates how administrators can centrally enforce security restrictions without configuring each computer individually.

Result: Standard users were unable to access Control Panel.

Account Lockout Demonstration

To validate the Account Lockout Policy, I intentionally entered an incorrect password multiple times using a standard domain account.

After exceeding the configured threshold, Active Directory automatically locked the account, preventing additional authentication attempts until an administrator unlocked the account.

This demonstrates how Group Policy helps protect enterprise environments from password guessing and brute-force attacks.

Result: Account locked after repeated failed authentication attempts.

DNS Configuration

For Active Directory to function correctly, client computers must use the Domain Controller as their preferred DNS server.

I configured the client workstation to point directly to the Domain Controller's IP address and verified successful name resolution using nslookup. Correct DNS configuration is essential because domain authentication, Group Policy processing, and locating network resources all rely on DNS.

Domain Join & Authentication

After verifying DNS connectivity, the Windows client computer was joined to the company.local domain.

Joining the domain allows users to authenticate using centralized Active Directory credentials instead of local computer accounts. This enables administrators to apply security policies, manage user access, and control devices from a single location.

After joining the domain, IT department users successfully signed into the workstation using their Active Directory credentials, confirming that authentication, DNS, and Group Policy were functioning correctly.

Key Takeaway

  • Deployed a Windows Server 2022 Domain Controller using AD DS.
  • Designed a department-based OU structure with Security Groups using RBAC principles.
  • Configured Group Policies to centrally manage security settings, verified using gpupdate and gpresult.
  • Configured DNS to support domain authentication.
  • Joined client computers to the domain and verified authentication.
