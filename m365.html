Identity & Access Management with Microsoft Entra ID

While exploring the Microsoft 365 Admin Center, I came across the Security Reader role, which got me curious about how access control actually works behind the scenes. That curiosity turned into this hands-on lab in Microsoft Entra ID, where I built out a small IT environment to see how role-based access control (RBAC), least privilege, and activity monitoring work together to keep an identity environment secure, and what happens when something as simple as a forgotten password needs to be resolved without over-granting access.

Role Design & Setup

I started by creating all the users through the Microsoft 365 Admin Center, requiring them to set a password when they first log in.

Administrator

Manages users, roles, and access configuration.

Helpdesk Technician

Handles password resets and user support.

Security Analyst

Monitors sign-in activity and investigates events.

End User

Standard account with limited access.

I created four users with defined responsibilities to simulate a real IT environment. Each role was assigned only the permissions required to perform their job, no more, no less. Jordan, the Helpdesk Technician, can reset passwords, but can't create new roles or touch security policies. Amelia, the Security Analyst, can view sign-in and audit logs, but can't modify user accounts.
This is least privilege in practice: giving each person exactly enough access to do their job, and nothing that would let a single compromised or careless account put the whole environment at risk.

Dynamic Group Membership

Through Entra ID, I created departmental groups using dynamic membership, automatically assigning users based on their Department attribute rather than adding them manually. For example, anyone tagged with the Security department is automatically included in the Security group, reducing the manual work of managing group membership as the organization grows.

Security Analyst Perspective: Sign-In Logs

As mentioned earlier, I wanted to understand how a role like Security Reader actually works behind the scenes. So Amelia, the Security Analyst, was assigned this role, giving her read-only access to sign-in and audit logs, enough to review authentication activity, but no ability to change any user or security settings. Her job in this scenario was to review the sign-in logs and confirm what actually happened during Sofia's login issue.

  • Multiple failed login attempts by Sofia
  • Successful authentication by Jordan (Helpdesk)
  • Successful login by Sofia after her password reset
  • Logs included timestamps, device details, and location for each attempt

The failure log showed: "Error validating credentials due to invalid username or password." This confirmed the issue was incorrect credentials, not a system or service failure, and matched exactly what Jordan had already diagnosed at the helpdesk level.

Audit Logs & Activity Tracking

Before contacting the helpdesk, Sofia had actually tried to reset her own password through self-service password reset (SSPR), but the attempt failed. Since SSPR requires a verified method like a phone number or authenticator app, and hers wasn't fully set up, she wasn't able to resolve it on her own and needed Jordan's help instead.

  • Failed self-service password reset attempt by Sofia
  • Password reset performed by Jordan (Helpdesk)
  • Target user: Sofia Diaz
  • Successful password update and login

All actions were fully traceable. Every step, from Sofia's failed SSPR attempt to Jordan's reset, was recorded in the audit logs, giving Amelia full visibility into the incident from start to finish. After reviewing the trail, she confirmed there was no suspicious activity involved, just credentials issue, and closed out the security side of the incident.

Key Takeaway

  • Designed four distinct roles (Admin, Helpdesk, Security Analyst, End User) using the principle of least privilege, ensuring no one had more access than their job required.
  • Simulated a real support incident, an end user locked out of her account, resolved through proper identity verification rather than assumptions.
  • Used sign-in and audit logs to confirm the root cause was a credentials issue, not a system failure, and to fully trace every action taken during the incident.
  • Applied the Security Reader role to give the Security Analyst read-only visibility into logs, without any ability to modify users or settings.
  • Next phase: implement Conditional Access policies and Multi-Factor Authentication (MFA) to catch and stop risky sign-in attempts as they happen.
