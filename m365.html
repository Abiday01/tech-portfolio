Microsoft Entra ID & Microsoft 365 Role-Based Access Control

While exploring the Microsoft 365 Admin Center, I discovered the Security Reader role, which led me to build a hands-on lab in Microsoft Entra ID. This project demonstrates how role-based access control (RBAC), least privilege, and monitoring work together in a real-world identity environment.

Role Design & Setup

I created four users with defined responsibilities to simulate a real IT environment. Each role was assigned only the permissions required to perform their job.

Administrator

Manages users, roles, and access configuration.

Helpdesk Technician

Handles password resets and user support.

Security Analyst

Monitors sign-in activity and investigates events.

End User

Standard account with limited access.

I started by creating all the users through the Microsoft 365 Admin Center, requiring them to set a password when they first log in.

Least Privilege in Practice

Each role was restricted to only the permissions required, reinforcing the principle of least privilege.

  • Helpdesk (Jordan) can reset passwords but cannot assign roles
  • Security Analyst (Amelia) can view logs but cannot modify users
  • End User (Sofia) can only access their own account

Through Entra ID, I created departmental groups using dynamic membership, allowing users to be automatically assigned based on their departments.

Password Reset by Helpdesk

After verifying the user’s identity, the helpdesk technician (Jordan) performed a secure password reset through Microsoft Entra ID. This restored access while maintaining proper security controls.

Security Analyst Perspective: Sign-In Logs

I analyzed sign-in logs to track authentication activity and identify issues.

  • Multiple failed login attempts by Sofia
  • Successful authentication by Jordan (Helpdesk)
  • Successful login by Sofia after password reset
  • Logs included timestamps, device details, and location

The failure log showed: "Error validating credentials due to invalid username or password." This confirmed the issue was incorrect credentials rather than a system failure.

Audit Logs & Activity Tracking

  • Failed self-service password reset attempt by Sofia
  • Password reset performed by Jordan (Helpdesk)
  • Target user: Sofia Diaz
  • Successful password update and login

All actions were fully traceable. Every activity performed by both Sofia and Jordan was recorded in audit logs, providing full visibility into the incident.

Key Takeaway

This project demonstrated how role separation and least privilege work together with logging and monitoring in a real-world identity system.

Even a simple login issue can be tracked end-to-end, allowing administrators and security analysts to validate actions, understand user behavior, and maintain security visibility.

In the next phase, I will enhance this environment by implementing Conditional Access policies and Multi-Factor Authentication (MFA).
