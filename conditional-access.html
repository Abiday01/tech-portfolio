Conditional Access & Zero Trust Security

In this lab, I designed and implemented Conditional Access policies in Microsoft Entra ID to secure a cloud-based identity environment. The objective was to enforce Zero Trust principles by applying risk-based authentication controls, strengthening identity protection, and analyzing authentication behavior through security logs.

Objectives

Before Conditional Access was implemented, users could sign in with only a valid username and password from any device or location. While passwords provide an initial layer of security, compromised credentials remain one of the most common attack vectors against cloud identity environments.

The goal of this lab was to strengthen identity security by implementing Microsoft Entra ID Conditional Access policies that evaluate factors such as user role, authentication strength, sign-in location, and risk before granting access. This approach follows Microsoft's Zero Trust security model, where every sign-in request is continuously verified rather than automatically trusted.

Environment & Roles

Sofia (Standard User)

Standard business user protected by baseline Conditional Access policies, including Multi-Factor Authentication (MFA).

Sofia was used to verify that the default security policies applied to every employee account. Since most users in an organization don't have elevated permissions, protecting these everyday accounts is the first line of defense against compromised credentials.

Jordan (IT Operations)

IT administrator with elevated permissions protected by layered Conditional Access policies.

Because Jordan performs administrative tasks, I applied additional controls beyond standard MFA, including Authentication Strength requirements and a three-hour sign-in frequency. These extra protections reduce the risk of unauthorized administrative access if credentials are compromised.

Amelia (Global Administrator)

Highest-privileged account protected with phishing-resistant authentication.

Amelia represents a Global Administrator with full control of the Microsoft Entra ID tenant. Since compromising this account could impact the entire organization, I required phishing-resistant MFA to provide stronger protection than traditional password and MFA alone.

Conditional Access Policy Design

I designed separate Conditional Access policies based on each user's role and level of privilege. Rather than applying identical controls to every account, policies were layered to provide stronger protection for administrative accounts while maintaining usability for standard users.

All Users

Required Multi-Factor Authentication (MFA) for every cloud sign-in, adding an additional layer of identity verification beyond passwords.

IT Operations

Required an Authentication Strength policy together with a 3-hour sign-in frequency. This ensured IT staff could only authenticate using approved MFA methods while reducing the risk of long-lived privileged sessions.

Global Administrator

Required phishing-resistant MFA to protect the highest privileged account. This policy allows only phishing-resistant authentication methods, such as FIDO2 Security Keys or Windows Hello for Business, providing stronger protection against credential theft and phishing attacks.

Organization Protection

Blocked legacy authentication protocols and configured location-based Conditional Access policies to reduce the attack surface and help prevent unauthorized access attempts.

Report-only mode

Before enforcing each Conditional Access policy, I first deployed it using Microsoft's Report-only mode. This allowed me to validate how policies would affect users without actually blocking sign-ins, helping identify unintended lockouts before enabling enforcement.

Testing & Log Analysis

MFA Enforcement

Standard user Sofia was prompted for MFA under the "All Users" policy, with successful authentication confirmed in sign-in logs.

Policy Layering

IT user Jordan had multiple Conditional Access policies applied simultaneously, including MFA and session controls.

Session Control

Re-authentication was enforced every 3 hours based on configured sign-in frequency.

Privileged Access Protection

Admin access was blocked when phishing-resistant MFA requirements were not met, preventing unauthorized elevation.

Geographic Restriction

Sign-in attempts from locations outside the United States were blocked based on location-based policy.

Sign-in Logs

  • Analyzed authentication success and failure events
  • Reviewed Conditional Access policies applied per login
  • Observed session token behavior and reduced MFA prompts

Audit Logs

  • Tracked identity and access management changes
  • Verified administrative actions
  • Confirmed policy deployment and enforcement

Key Takeaway

  • Rolling policies out in Report-only mode first let me verify how they would behave before enforcing them, helping prevent accidental user lockouts during deployment.
  • Privileged accounts need stronger protection than standard users. Requiring phishing-resistant authentication for Amelia's Global Administrator account reduced the risk of credential theft leading to full administrative access.
  • Blocking legacy authentication significantly reduces the attack surface
  • Reviewing both sign-in logs and audit logs provided a complete picture of policy enforcement, allowing authentication events and administrative changes to be verified throughout the deployment.
  • Zero Trust isn't a single security feature. It comes from layering multiple controls Conditional Access, MFA, Authentication Strength, location policies, and continuous monitoring to verify every sign-in before access is granted.
