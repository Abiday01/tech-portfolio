Conditional Access & Identity Security (Microsoft Entra ID)

In this lab, I designed and implemented Conditional Access policies in Microsoft Entra ID to secure a cloud-based identity environment. The objective was to enforce Zero Trust principles by applying risk-based authentication controls, strengthening identity protection, and analyzing authentication behavior through security logs.

Objectives

  • Design and implement Conditional Access policies based on identity risk and role sensitivity
  • Enforce multi-factor authentication (MFA) and advanced authentication strength policies
  • Apply Role-Based Access Control (RBAC) and least privilege principles
  • Validate authentication flows across different user personas
  • Analyze sign-in and audit logs for security monitoring and investigation

Environment & Roles

Sofia (Standard User)

Baseline access used to validate default Conditional Access enforcement, basic MFA policy

Jordan (IT Operations)

Elevated access with layered policies and session controls

Amelia (Privileged Administrator)

High-risk account requiring phishing-resistant authentication

Conditional Access Policy Design

All Users: Enforced multi-factor authentication (MFA)

IT Operations: MFA + session controls (sign-in frequency: 3 hours)

Privileged Administrators: Required phishing-resistant MFA (authentication strength)

All Users: Blocked legacy authentication (POP/IMAP and non-modern protocols)

All Users: Blocked high-risk sign-ins based on identity protection signals

Location-Based Policy: Access blocked from all countries except the United States

Testing & Log Analysis

  • MFA Enforcement: Standard user Sofia was prompted for MFA under the “All Users” policy, with successful authentication confirmed in sign-in logs
  • Policy Layering: IT user Jordan had multiple Conditional Access policies applied simultaneously, including MFA and session controls
  • Session Control: Re-authentication was enforced every 3 hours based on configured sign-in frequency
  • Privileged Access Protection: Admin access was blocked when phishing-resistant MFA requirements were not met, preventing unauthorized elevation
  • Geographic Restriction: Sign-in attempts from locations outside the United States were successfully blocked based on location-based policy

Sign-in Logs:

  • Analyzed authentication success and failure events
  • Reviewed Conditional Access policies applied per login
  • Observed session token behavior and reduced MFA prompts

Audit Logs:

  • Tracked identity and access management changes
  • Verified administrative actions
  • Confirmed policy deployment and enforcement

Key Takeaways

  • Conditional Access enforces Zero Trust by continuously validating access decisions
  • Multiple Conditional Access policies can apply to a single user, with stricter controls taking priority
  • Privileged accounts require stronger authentication controls and tighter enforcement
  • Blocking legacy authentication significantly reduces the attack surface
  • Location-based restrictions help prevent unauthorized access attempts
  • Effective identity security requires both preventive controls and continuous monitoring
