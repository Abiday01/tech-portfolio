Conditional Access & Zero Trust Security

In this lab, I designed and implemented Conditional Access policies in Microsoft Entra ID to secure a cloud-based identity environment. The objective was to enforce Zero Trust principles by applying risk-based authentication controls, strengthening identity protection, and analyzing authentication behavior through security logs.

Objectives

Before Conditional Access was implemented, any user with valid credentials could attempt to sign in from virtually any location or device. Although passwords alone provide one layer of protection, compromised credentials remain one of the most common attack vectors against cloud identity environments.

The goal of this lab was to strengthen identity security by introducing Conditional Access policies that continuously evaluate sign-in risk, user role, authentication strength, and location before access is granted, following Microsoft's Zero Trust security model.

Environment & Roles

Sofia (Standard User)

Baseline access used to validate default Conditional Access enforcement, basic MFA policy

Sofia represents a typical end user with standard business access. Since most breaches start with an everyday account being compromised, she was used to validate that baseline protections like MFA apply to everyone, not just privileged accounts.

Jordan - IT Operations administrator.

Elevated access with layered policies and session controls

This role performs privileged administrative work, additional controls such as MFA and session restrictions were applied to reduce the impact of compromised credentials.

Amelia (Privileged Administrator)

High-risk account requiring phishing-resistant authentication

Amelia represents a Global Administrator with the highest level of access in the tenant. Since a compromised admin account could affect the entire organization, phishing-resistant authentication was required for her specifically, not just password + MFA, to prevent credential theft from leading to full environment compromise.

Conditional Access Policy Design

I designed separate Conditional Access policies based on each user's role and level of privilege. Rather than applying identical controls to every account, policies were layered to provide stronger protection for administrative accounts while maintaining usability for standard users.

All Users

Required Multi-Factor Authentication (MFA) for every cloud sign-in, adding an additional layer of identity verification beyond passwords.

IT Operations

Required an Authentication Strength policy together with a 3-hour sign-in frequency. This ensured IT staff could only authenticate using approved MFA methods while reducing the risk of long-lived privileged sessions.

Global Administrator

Required phishing-resistant MFA to protect the highest privileged account. This policy allows only phishing-resistant authentication methods, such as FIDO2 Security Keys or Windows Hello for Business, providing stronger protection against credential theft and phishing attacks.

Organization Protection

Blocked legacy authentication protocols and configured location-based Conditional Access policies to reduce the attack surface and help prevent unauthorized access attempts.

Report-only mode

Before enforcing each Conditional Access policy, I first deployed it using Microsoft's Report-only mode. This allowed me to validate how policies would affect users without actually blocking sign-ins, helping identify unintended lockouts before enabling enforcement.

Testing & Log Analysis

MFA Enforcement

Standard user Sofia was prompted for MFA under the "All Users" policy, with successful authentication confirmed in sign-in logs.

Policy Layering

IT user Jordan had multiple Conditional Access policies applied simultaneously, including MFA and session controls.

Session Control

Re-authentication was enforced every 3 hours based on configured sign-in frequency.

Privileged Access Protection

Admin access was blocked when phishing-resistant MFA requirements were not met, preventing unauthorized elevation.

Geographic Restriction

Sign-in attempts from locations outside the United States were blocked based on location-based policy.

Sign-in Logs

  • Analyzed authentication success and failure events
  • Reviewed Conditional Access policies applied per login
  • Observed session token behavior and reduced MFA prompts

Audit Logs

  • Tracked identity and access management changes
  • Verified administrative actions
  • Confirmed policy deployment and enforcement

Key Takeaway

  • Designing Conditional Access around roles, not just individual users, made it possible to apply stronger controls to Jordan and Amelia without adding friction for Sofia's day-to-day access.
  • Testing policies in Report-only mode first showed me how they'd behave before enforcing them, which is the safer way to roll out access changes without risking an unintended lockout.
  • Privileged accounts need more than MFA. Requiring phishing-resistant authentication for Amelia's Global Administrator account reflects how much more damage a compromised admin account can do compared to a standard user.
  • Blocking legacy authentication closed off an entire category of attack that MFA alone wouldn't have caught, since older protocols don't support it.
  • Sign-in and audit logs gave me two different views of the same incident, authentication behavior on one side, administrative actions on the other, which together made the whole policy rollout traceable.
  • Zero Trust isn't a single setting, it's the result of several layered policies working together, each one closing a different gap the others didn't cover.
